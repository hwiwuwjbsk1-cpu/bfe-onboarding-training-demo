# 十二条令三阶段互动训练项目 PRD

## 1. 项目背景

当前新人培训 demo 的主流程为：

1. 翻开新人通关手册。
2. 阅读培训 HTML 知识页。
3. 进入第一层轻量小游戏。
4. 完成或跳过第一层小游戏后，弹出“是否进入下一阶段”。
5. 点击“进入下一阶段”后，继续后续培训互动任务。

本项目需要放在图一弹窗之后，也就是用户点击“进入下一阶段”后展示。

注意：本项目不是替换原来的 demo，也不是重做整个培训系统，只是在现有流程中新增一个“十二条令三阶段训练”模块。

## 2. 项目目标

用轻松、有互动感的方式，让新员工逐步理解并掌握“十二条令”。

十二条令包括：

1. 确认指令
2. 及时汇报
3. 全员周报
4. 说到做到
5. 保持准时
6. 解决问题
7. 日清消息
8. 会议记录
9. 事事备忘
10. 三条总结
11. 一页报告
12. 统计分析

训练分为三次完成，每次进入只能完成一个阶段。

## 4. 核心用户流程

### 第一次进入

用户完成第一层小游戏后，点击“进入下一阶段”。

进入：

```text
十二条令训练营 · 第一阶段
```

第一阶段完成后：

- 显示阶段完成弹窗。
- 告诉用户“本阶段已完成”。
- 按钮为“继续培训项目”。
- 本次不直接展示第二阶段。
- 下次再进入该模块时，显示第二阶段。

### 第二次进入

展示：

```text
十二条令训练营 · 第二阶段
```

第二阶段完成后：

- 显示阶段完成弹窗。
- 本次不直接展示第三阶段。
- 下次进入时，显示第三阶段。

### 第三次进入

展示：

```text
十二条令训练营 · 第三阶段
```

第三阶段完成后：

- 标记整个十二条令训练完成。
- 允许继续进入原培训项目后续流程。
- 不影响原来的入职地图或行走问答流程。

## 5. 三阶段玩法设计

### 第一阶段：找出十二条令

目标：让用户知道十二条令具体包含哪些内容。

玩法：

- 页面展示多个可点击卡片。
- 其中有 12 个正确条令，也有若干干扰项。
- 用户需要选出属于十二条令的项目。
- 每选对一个卡片高亮。
- 选错卡片轻微抖动或变灰。
- 选满 12 个正确项后通关。

正确项：

- 确认指令
- 及时汇报
- 全员周报
- 说到做到
- 保持准时
- 解决问题
- 日清消息
- 会议记录
- 事事备忘
- 三条总结
- 一页报告
- 统计分析

干扰项示例：

- 每天加班
- 所有问题自己扛
- 只听领导安排
- 会议越长越好
- 不用记录，靠记忆
- 先做再说，不用确认

通关条件：

- 正确选择全部 12 个条令。

失败机制：

- 不做强失败。
- 选错只提示，不扣除进度。
- 保持轻松，避免像考试系统。

### 第二阶段：判断靠谱行为

目标：让用户理解哪些行为符合十二条令。

玩法形式：

- 单选题
- 判断题
- 填空式选择题

建议做 6 到 8 道题，每题展示一个职场场景，用户选择最匹配的条令或补全关键行为。

示例题：

1. 主管临时安排任务后，你先复述目标、交付物、截止时间，并确认是否理解正确。
   - 正确答案：确认指令

2. 会议结束后，把结论、关键讨论过程、行动计划、负责人和 ddl 写出来并同步给参会人。
   - 正确答案：会议记录

3. 工作中发现问题比预期复杂，自己尝试后仍无法推进，于是及时说明进展、卡点和需要的支持。
   - 正确答案：及时汇报

4. 汇报方案时先给结论，再列理由，最后给下一步建议。
   - 正确答案：三条总结 / 一页报告

5. 约定 11:00 开会，提前完成人员集合、设备调试、PPT 调整，11:00 正式进入会议议程。
   - 正确答案：保持准时

通关条件：

- 完成全部题目。
- 答对率达到 70% 以上。
- 如果未达到，可以允许重新答当前阶段。

### 第三阶段：完成一篇会议纪要

目标：让用户实际产出一篇会议纪要。

玩法：

页面给出一个模拟会议场景：

```text
会议主题：新员工培训流程优化
参会人：人事、业务导师、仓储负责人、新员工代表
讨论内容：
1. 新员工希望培训资料更清晰。
2. 业务导师建议增加小游戏互动。
3. 仓储负责人提出要增加国内仓流程介绍。
4. 人事确认下周五前完成新版培训材料初稿。
```

用户需要填写会议纪要结构：

1. 会议主题
2. 会议时间
3. 参会人
4. 会议结论
5. 关键讨论过程
6. 行动计划
7. 负责人
8. ddl 时间

校验规则：

- 不需要 AI 评分。
- 用基础规则判断是否完成。
- 至少填写会议主题、会议结论、行动计划、负责人、ddl。
- 总字数达到 80 字以上即可通关。

完成后显示：

- “会议纪要已完成”
- 展示用户填写的内容
- 给出标准结构提示

## 6. 进度规则

使用 localStorage 保存进度，不需要后端。

建议 key：

```ts
const TWELVE_RULES_STAGE_KEY = 'bfe-twelve-rules-stage-v1';
const TWELVE_RULES_SESSION_KEY = 'bfe-twelve-rules-session-lock-v1';
```

进度类型：

```ts
type TwelveRulesStage = 1 | 2 | 3 | 4;
```

含义：

- `1`：第一阶段待完成
- `2`：第二阶段待完成
- `3`：第三阶段待完成
- `4`：全部完成

关键规则：

1. 用户每次进入模块，只能完成当前阶段。
2. 当前阶段完成后，localStorage 阶段 +1。
3. 本次页面只显示完成弹窗，不自动进入下一阶段。
4. 用户下次重新进入该模块时，才显示下一阶段。
5. 如果用户刷新页面，可以继续当前阶段。
6. 如果阶段已完成到 4，点击进入时显示“十二条令训练已完成”，并允许继续后续培训流程。

## 7. 页面位置与跳转逻辑

当前图一弹窗为：

```text
是否进入下一阶段？
按钮：进入下一阶段
```

新逻辑：

```text
点击“进入下一阶段”
-> 进入 TwelveRulesTraining 页面
-> 完成当前阶段
-> 显示阶段完成弹窗
-> 点击“继续培训项目”
-> 再执行原来的 completeMiniLayer / 进入后续流程逻辑
```

也就是说，新模块插入在“进入下一阶段”和“继续原培训流程”之间。

## 8. 技术栈

基于现有项目继续开发。

推荐技术栈：

- React
- TypeScript
- Vite
- CSS 原生样式
- lucide-react 图标
- localStorage 保存进度
- 静态资源部署，无后端

不要新增复杂依赖。

不建议使用：

- 后端数据库
- 登录系统
- AI 评分接口
- 大型 UI 框架
- 复杂游戏引擎

## 10. UI 风格要求

整体风格要与现有小游戏页面一致：

- 清新
- 蓝白色为主
- 办公培训感
- 不要太企业宣传片
- 不要强行塞太多企业口号
- 不要像考试系统
- 不要出现开发者内部词，比如 `miniLayer`、`前置小游戏`、`原流程`

页面结构建议：

左侧：

- 阶段标题
- 当前任务说明
- 进度条
- 十二条令主题视觉

右侧：

- 当前阶段游戏主体

完成弹窗：

- 标题：`第一阶段完成`
- 副标题：`你已识别十二条令的核心内容`
- 按钮：`继续培训项目`
- 可显示下一次提示：`下次进入将开启第二阶段`

## 12. 验收标准

开发完成后需要满足：

1. 原来的书本开场、资料 HTML、第一层小游戏不被破坏。
2. 第一层小游戏完成后，点击“进入下一阶段”进入十二条令训练。
3. 第一次只显示第一阶段。
4. 第一阶段完成后，不直接进入第二阶段。
5. 第二次进入才显示第二阶段。
6. 第三次进入才显示第三阶段。
7. 第三阶段完成后，能继续原培训项目后续流程。
8. 结果弹窗里不要再出现多余的积分规则提示块。
9. 页面不需要上下大幅滚动。
10. 构建命令通过。

验收命令：

```bash
npm run build
npx tsc --noEmit
```

## 13. 开发优先级

### P0

- 三阶段页面
- 每次只完成一个阶段
- localStorage 进度
- 接入“进入下一阶段”按钮
- 完成后回到原培训流程

### P1

- 更好的动画
- 卡片翻转 / 点击反馈
- 会议纪要完成后的结构化预览
- 阶段完成弹窗美化

### P2

- 重置训练进度按钮
- 管理员查看完成情况
- 导出会议纪要
- 更复杂的评分机制

## 14. 给开发模型的特别提醒

不要重做项目。

不要改原来的书本开场。

不要改培训 HTML 阅读页。

不要替换第一层小游戏。

只在“进入下一阶段”之后插入这个三阶段模块。

所有图片优先引用现有 `public/assets/lessons/` 和 `public/assets/game/` 目录中的图片，不要随意新建不一致风格的图片。
