# 组织架构协作导航游戏 PRD

## 1. 项目背景

当前新人培训 HTML 中已经整理了组织架构相关内容，包括：

- 集团主体
- 事业部划分
- 核心品牌
- 财务对接人
- 投资背景

用户提供的真实组织架构截图显示，当前组织树以“肖友泉”为顶层，下设：

- 战略委员会
- 总裁办
- 财务部
- 技术部
- 业务发展
- 产品及运营
- 其他（待设置部门）

本项目希望基于这些信息设计一款小游戏，帮助新员工理解组织结构、部门归属和跨部门协作路径。

注意：这个游戏不是通讯录，也不是完整 HR 系统。它是新人培训中的轻量互动任务，用来建立“知道公司有哪些关键部门、遇到事情大概找谁”的基础认知。

## 2. 项目目标

让新人通过游戏完成三件事：

1. 认识公司组织结构中的主要部门。
2. 理解集团主体、事业部、品牌和对接关系。
3. 在常见工作场景中判断应该找哪个部门或走怎样的协作路径。

游戏体验要轻松、清新、可重复，每次进入题目可以随机变化，不要求一次背完整组织架构。

## 3. 游戏名称

推荐名称：

```text
组织导航局
```

备选名称：

- 部门寻路
- 协作导航站
- 组织架构探图
- 新人找部门

建议使用“组织导航局”，因为它更像游戏任务，不像考试。

## 6. 游戏定位

游戏类型：

```text
轻量益智 + 场景派单 + 路径拼接
```

核心体验：

- 不让新人死记硬背组织架构。
- 通过拖拽、选择和路径连接，理解部门之间的关系。
- 每次随机抽取一部分部门和场景，不要求一次覆盖全部。

单次体验时长：

```text
3-5 分钟
```

适合接入位置：

- 可以放在“十二条令训练”之后，作为下一段培训互动任务。
- 也可以作为入职地图里的一个独立节点：`组织架构 / 找部门`。

## 7. 核心玩法概览

游戏分为三关，但一次完整游戏可以在同一次内完成三关。和十二条令项目不同，本游戏不强制隔天或下次解锁。

如果希望后续做成长线版本，也可以改成每次只完成一关。

### 第一关：部门归位

目标：认识组织树中的主要部门。

玩法：

- 页面左侧展示空白组织树。
- 顶层固定显示“肖友泉”。
- 下方出现若干部门卡片。
- 用户把部门卡片拖到组织树的正确位置。
- 每次随机出现 4 到 6 个部门，不要求一次覆盖全部。

可出现部门：

- 战略委员会
- 总裁办
- 财务部
- 技术部
- 业务发展
- 产品及运营
- 其他（待设置部门）

正确逻辑：

- 这些部门均为“肖友泉”下一级。
- 第一版不需要做更深层子部门。

通关条件：

- 当前轮次所有部门归位正确。

错误反馈：

- 放错位置时卡片轻微回弹。
- 显示一句提示：`这个部门属于同一层级，先放回主组织树。`

### 第二关：业务板块配对

目标：理解 HTML 中整理的集团主体、事业部和核心品牌关系。

玩法：

- 页面展示三个区域：`事业部`、`核心品牌`、`财务对接`。
- 用户需要把卡片配对。
- 每次随机抽取 2 到 3 组，不要求一次全部记住。

基础数据：

```text
物流事业部 -> 出口易 -> 陈滢 / 陈丽纯 / 莫何冰 / 林静欣
SAAS事业部 -> 亚速维亿 -> 林静欣
DM项目 -> 广州斗满 / 法国DM / 德国DM -> 林静欣
```

玩法形式建议：

- 连线配对
- 拖拽配对
- 三列选择

推荐第一版使用“三列选择”，开发成本低：

1. 给出一个事业部。
2. 用户选择对应核心品牌。
3. 用户选择对应财务对接。

通关条件：

- 完成当前随机题组。

错误反馈：

- 选错不失败，只提示正确组合。
- 错题进入本地错题本。

### 第三关：协作派单

目标：让新人学会遇到事情找哪个部门或形成怎样的协作路径。

玩法：

- 系统给出一个工作场景。
- 用户从部门卡片中选择“第一联系人”或“协作路径”。
- 每次随机 3 个场景。

示例场景：

#### 场景 1：报销和发票

```text
你需要确认报销规则、发票要求和合同审批流程，应该优先找哪个部门？
```

正确答案：

```text
财务部
```

#### 场景 2：系统问题

```text
内部系统出现登录异常，影响工作推进，应该优先找哪个部门协助排查？
```

正确答案：

```text
技术部
```

#### 场景 3：产品功能反馈

```text
客户反馈后台报表导出慢，希望优化功能体验，应该优先关联哪个部门？
```

正确答案：

```text
产品及运营
```

#### 场景 4：新业务机会

```text
客户提出一个新的合作模式，需要判断是否值得推进和由谁跟进，应该优先关联哪个方向？
```

正确答案：

```text
业务发展
```

#### 场景 5：跨部门重大事项

```text
一个事项涉及多个部门资源协调，需要更高层统一判断和协调。
```

可接受答案：

```text
总裁办
战略委员会
```

说明：

- 总裁办偏执行协调。
- 战略委员会偏重大方向判断。

通关条件：

- 完成 3 个随机场景。
- 正确率达到 70% 以上即可通关。
- 未达到则允许重玩第三关。

## 8. 页面结构

### 游戏大厅

页面内容：

- 标题：`组织导航局`
- 副标题：`看懂公司结构，找到协作路径`
- 左侧：组织树简图
- 右侧：任务卡片
- 主按钮：`开始导航`

可见信息：

```text
本轮随机抽取部门和场景，每次练习都会不一样。
```

不要出现：

- “考试”
- “背诵”
- “内部测试”
- “组织架构考核”

### 游戏主界面

布局建议：

- 左侧：当前关卡说明和组织树
- 右侧：交互区域
- 顶部：进度条
- 底部：确认按钮 / 下一题按钮

视觉风格：

- 蓝白清新
- 文件夹/组织树元素可以保留，但不要像传统 OA 后台
- 卡片圆角不超过 8px
- 信息密度适中，不要大段文字

### 完成弹窗

标题：

```text
组织导航完成
```

文案：

```text
你已经了解主要部门、业务板块和常见协作路径。
```

按钮：

```text
继续培训项目
再玩一次
```

## 10. 进度与随机规则

使用 localStorage 保存游戏完成状态和错题。

建议 key：

```ts
const ORG_GAME_PROGRESS_KEY = 'bfe-org-game-progress-v1';
const ORG_GAME_MISTAKES_KEY = 'bfe-org-game-mistakes-v1';
```

随机规则：

1. 第一关每次随机抽取 4 到 6 个部门。
2. 第二关每次随机抽取 2 到 3 组事业部关系。
3. 第三关每次随机抽取 3 个场景。
4. 不要求一次覆盖全部组织架构。
5. 用户重玩时题目重新随机。

进度规则：

- 完成游戏后记录 `completed: true`。
- 可重复游玩。
- 重玩不影响原培训主线。
- 如果接入培训主线，完成后向父页面发送完成消息。

## 11. 嵌入方式

如果沿用当前十二条令项目的独立 HTML 模式，建议新增：

```text
org-structure-game.html
```

接入方式：

```text
第一层小游戏 / 十二条令训练完成
-> 打开 org-structure-game.html?embed=1
-> 完成组织导航局
-> postMessage 通知父页面
-> 继续原培训项目
```

建议消息类型：

```ts
const ORG_GAME_COMPLETE = 'bfe-org-game-complete';
const ORG_GAME_CANCEL = 'bfe-org-game-cancel';
```

如果使用 React 方式，也可以做成组件：

```text
src/features/org-game/OrgStructureGame.tsx
```

但第一版建议独立 HTML，开发成本低，便于小模型单独实现。

## 12. 技术栈

推荐：

- HTML
- CSS
- 原生 JavaScript
- localStorage
- postMessage

或在现有 React 项目中使用：

- React

## 13. 验收标准

开发完成后需要满足：

1. 能独立打开 `org-structure-game.html`。
2. 能嵌入现有培训项目 iframe。
3. 游戏至少包含三关：部门归位、业务板块配对、协作派单。
4. 每次进入题目随机变化。
5. 不要求一次覆盖完整组织架构。
6. 完成后能显示完成弹窗。
7. 嵌入模式下完成后能通知父页面继续培训流程。
8. UI 风格和现有小游戏一致，清新、轻量、不像后台系统。
9. 不出现文字重叠、按钮被遮挡、页面大幅滚动。
10. 构建检查通过。

验收命令：

```bash
npm run build
npx tsc --noEmit
```

## 15. 给开发模型的提醒

不要把它做成通讯录。

不要让用户背完整组织架构。

不要一次塞太多部门。
